01 · Why

Why prompt engineering matters for QA

QA work is uniquely well-suited to LLMs — and uniquely vulnerable to bad output.

⚡ Where LLMs help
  • Test case generation from requirements
  • Bug report drafting & triage
  • Test data synthesis
  • Log and stacktrace analysis
  • Selenium / Playwright / API test boilerplate
⚠️ Where they hurt you
  • Hallucinated APIs and methods
  • Missed edge cases and negative paths
  • Confidently wrong assertions
  • Subtle off-by-one bugs in generated code
  • Coverage illusion — many tests, low value

Prompt engineering is the difference between the two columns.

02 · Anatomy

Anatomy of a prompt

Source: ISTQB® CT-GenAI v1.1 — Testing with Generative AI

The official ISTQB Certified Tester — Testing with Generative AI syllabus defines a prompt as a structure with six components. Hover each block for the definition.

1
ROLE
Perspective or persona the model should take (tester, test manager, automation engineer)
2
CONTEXT
Background — test object, functionality, environment, conditions
3
INSTRUCTION
The directive — clear, imperative, concise. What to do.
4
INPUT DATA
User stories, ACs, screenshots, code, existing tests, examples
5
CONSTRAINTS
Restrictions and special considerations the model must respect
6
OUTPUT FORMAT
Expected shape — table, Gherkin, JSON, markdown, length

Example using all six components

You are a senior QA tester.                                  // 1. ROLE

We are testing a password-reset flow on a public web app.    // 2. CONTEXT
The link arrives by email, expires after 30 minutes,
and is single-use.

Generate manual test cases that cover every acceptance       // 3. INSTRUCTION
criterion below, including positive, negative, and
boundary scenarios.

User story: "As a registered user, I want to reset my        // 4. INPUT DATA
password via email link..."
Acceptance criteria:
- AC1: Email is sent only if the account exists.
- AC2: Same confirmation message appears regardless.
- AC3: Reset link expires after 30 minutes.
- AC4: Reset link is single-use.
- AC5: Password policy: 8+ chars, 1 number, 1 special.

Limit to 12 cases. Do not invent UI elements not             // 5. CONSTRAINTS
implied by the acceptance criteria. Each AC must
be covered by at least one positive and one negative case.

Output as a markdown table with columns:                     // 6. OUTPUT FORMAT
ID | Title | Preconditions | Steps | Expected Result | Priority | AC covered

Definitions quoted from the ISTQB® CT-GenAI v1.1 syllabus. Every block above maps to one of the six components defined there. The acronym R-C-I-I-C-O can be a memory aid (Role, Context, Instruction, Input data, Constraints, Output format).

03 · Principles

Five core principles

  1. Be specific. Vague prompts produce vague tests. "Test the login page" → generic happy path. "Test 8 scenarios for /login including lockout after 5 failures" → useful coverage.
  2. Provide context. Stack, schema, prior failures, acceptance criteria. The model can't guess what it doesn't know.
  3. Show examples. One good example beats a paragraph of description. Few-shot prompting works.
  4. Iterate. Treat the prompt like code — refine, version, peer-review. Save winners.
  5. Verify. Never ship LLM output unread. Especially assertions and test data.

Mnemonic: S-C-E-I-V — Specific, Context, Examples, Iterate, Verify.

04 · Techniques

Techniques in your toolbox

Zero-shot Just ask. Fast for simple tasks.
"List edge cases for a date picker."
Few-shot Provide 2–5 examples of input → desired output.
Best for consistent format across many requests.
Chain-of-thought "Think step by step before answering."
Reduces logic errors in test design and triage.
Role prompting "Act as a security tester / accessibility auditor."
Shifts the lens, surfaces different defects.
Constrained output Force JSON / schema / Gherkin.
Pipeline-friendly, machine-parseable.
Self-critique "Now review your answer for missing cases."
Two-pass = better coverage in practice.

06 · Compare

Vague vs. engineered — same task, different result

Prompt
Write some tests for the login page.
Likely output

Generic happy-path test for username/password. No edge cases, no framework specified, selectors that don't match your app, possibly fabricated APIs. Looks fine until you run it.

Prompt
You are a senior QA engineer using Playwright + TypeScript.
The login page is at /login with fields #email, #password,
button [data-testid="login-submit"]. Backend returns 401
on bad credentials and locks the account after 5 failures.

Generate 8 test cases covering: happy path, empty fields,
malformed email, lockout, password visibility toggle, and
Enter-key submission.

Output as a Playwright test file. Use page.getByTestId where
possible. Do not invent selectors not listed above.
Likely output

Runnable spec file with stable selectors, real edge cases, and assertions tied to documented backend behavior.

07 · Pitfalls

Common pitfalls (and fixes)

🌀
Hallucinated APIs. The model invents a method that sounds right.
Fix: provide real signatures in context. Always run the code.
🤐
Missing constraints. "Generate test data" → 10 plausible but invalid rows.
Fix: state schema, ranges, locales, required formats explicitly.
📏
Coverage illusion. 50 generated tests, all on the happy path.
Fix: ask for boundary, negative, and security cases by name.
🎭
Over-trusting tone. Confident wording does not equal correctness.
Fix: ask "what assumptions did you make?" as a second pass.
🔁
Stale context. Spec changes; prompt template doesn't.
Fix: version prompts in the same repo as your tests.

Suggested workflow — the 5-step practice loop

  1. Draft in the Lab. Write your prompt for one of the 4 challenges. Get it to 80%+.
  2. Send to AI Studio. Click 🚀 Test in AI Studio in the Lab — it copies your prompt and opens AI Studio in a new tab. Paste into the prompt field.
  3. Run and read. Run the prompt. Check whether the output actually solves the QA task: are the test cases real, are the selectors correct, is the JSON parseable?
  4. Compare with reference. Back in the Lab, open the reference prompt, run that in a new AI Studio chat, and compare both outputs.
  5. A/B across models. Click ⚔️ Test in LMSYS Arena to run the same prompt against two models side-by-side. If one fails badly, your prompt is brittle — tighten it.

Pro tips for AI Studio specifically

  • Set Temperature = 0.2 for QA work — you want deterministic, not creative.
  • Turn on Structured output with a JSON schema for any test data or triage task.
  • Use the "Get code" button to convert a working prompt into an API call you can put in CI.
  • Save winning prompts via "Save to library" — version-control the winners alongside your tests.
11 · Practice

Exercises — try these

Click each to reveal a sample answer.

1. Rewrite this prompt: "Generate test cases for the search bar."
Improved: 
You are a QA engineer. The search bar is on the homepage,
accepts up to 200 chars, supports quoted phrases and the
operators AND/OR/NOT. Backend trims whitespace and is
case-insensitive.

Generate 12 test cases covering: empty input, max length,
SQL/script injection, Unicode, operator precedence, and
no-results state.

Output as a Gherkin feature file.
2. The model invented a method page.waitForStableNetwork(). What went wrong?
Likely a hallucination — the real Playwright API is page.waitForLoadState('networkidle'). The prompt didn't constrain the model to documented APIs. Fix by including a snippet of the framework's real API surface or instructing: "Use only documented Playwright API methods. If unsure, say so."
3. You need test data for users in 5 locales. Sketch a prompt. 
Generate 10 user records as a JSON array. Each record:
{ id, name, email, phone, locale }.

Locales to cover (2 each): en-US, de-DE, ja-JP, ar-SA, pt-BR.

Constraints:
- Names must be plausible for the locale (not just translated).
- Phones must match locale dialing format.
- Emails must be valid RFC 5322.
- Include one edge case per locale (max-length name,
  RTL chars, etc.).

Output: valid JSON only, no commentary.
4. You got 50 test cases but they all pass on day one. Why might that be a problem?
Coverage illusion. The model likely generated only the happy path or trivial variations. Tests that always pass don't catch regressions. Add a self-critique pass: "Review the cases above. Which behaviors are NOT yet tested? Add cases for boundaries, error states, and concurrency." Then trim duplicates.
12 · Take home

Take this home

  • Build every prompt from the six ISTQB CT-GenAI components: Role · Context · Instruction · Input data · Constraints · Output format.
  • Show one good example before asking for many.
  • Force structured output (JSON / Gherkin) for anything pipeline-bound.
  • Always do a self-critique pass for coverage.
  • Never ship LLM-written assertions unread.
  • Version-control prompts alongside the tests they generate.
